1 Введение

1.1 Общая информация

Документ содержит руководство администратора по настройке обмена сообщениями между независимыми экземплярами брокера сообщений RabbitMQ с помощью технологии Federation. Руководство применимо для RabbitMQ, работающих под управлением ОС Windows и Linux. 

Технология Federation (Федерация) обеспечивает ретрансляцию сообщений, публикуемых в точках обмена (exchanges) или очередях (queues) одного экземпляра брокера сообщений в другой посредством восходящих потоков (upstream). Определение точек обмена и очередей, которые учувствуют в федеративном обмене, выполняется на основе политик (policies) RabbitMQ.

Восходящие потоки – односторонние, то есть передача информации выполняется только в одну сторону. Поэтому для включения в Федерацию двух экземпляров брокера сообщений необходимо на каждом экземпляре организовать восходящий поток до другого экземпляра и настроить политики ретрансляции.

Использование технологии Federation не требует включения экземпляров брокера сообщений в кластер и может использоваться совместно с технологий кластеризации RabbitMQ.

Технология Federation позволяет иерархически объединить в Федерацию множество экземпляров RabbitMQ для решения следующих задач:

ü  повышение производительности;

ü  повышение уровня отказоустойчивости брокера сообщений, как  единого сервиса очередей сообщений;

ü  организацию распределенной среды (территориально, функционально и т.д.) транспорта сообщений.

Технология Federation RabbitMQ реализована в виде двух плагинов (plugins):

ü  rabbitmq_federation – плагин поддержки технологии Federation;

ü  rabbitmq_federation_management – плагин администрирования Federation для веб-консоли RabbitMQ.

Настоящий документ содержит инструкции по активации плагинов, настройки восходящих потоков и политик, а также руководства по контролю работы информационного обмена между экземплярами RabbitMQ с помощью технологии Federation.

Более подробную информацию о технологии Federation RabbitMQ можно получить на сайте производителя по адресу: http://www.rabbitmq.com/federation.html.

1.2 Необходимые условия

Выполнена установка и запуск двух экземпляров RabbitMQ в соответствии со следующими руководствами:

ü  «Установка брокера сообщений  RabbitMQ для ОС Windows Server. Руководство администратора»;

ü  «Установка RabbitMQ для SLES11. Руководство администратора» или «Установка RabbitMQ для SLES12. Руководство администратора» или «Установка RabbitMQ для SLES12 SP3. Руководство администратора».

1.3 Необходимые знания

Для выполнения представленных в документе инструкций необходимо иметь:

ü  учетные данные пользователей ОС с правами администратора;

ü  адреса, имена хостов и используемые порты экземпляров RabbitMQ;

ü  пароли пользователей admin и rsdu, заведенных в RabbitMQ;

ü  опыт работы с веб-консолью управления RabbitMQ;

ü  базовые знания и навык администрирования ОС, используемых на серверах экземпляров RabbitMQ.

 

 

2 Активация плагинов Federation (plugins)

Технология Federation RabbitMQ реализуется двумя плагинами:

ü  rabbitmq_federation – плагин поддержки технологии Federation;

ü  rabbitmq_federation_management – плагин администрирования Federation для веб-консоли RabbitMQ.

Плагины Federation входят в базовый комплект плагинов брокера сообщений RabbitMQ.

Для активации плагинов Federation необходимо выполнить следующие команды от имени пользователя с правами администратора:

ü  Linux

rabbitmq-plugins enable rabbitmq_federation

ü  Windows

rabbitmq-plugins.bat enable rabbitmq_federation

и

ü  Linux

rabbitmq-plugins enable rabbitmq_federation_management

ü  Windows

rabbitmq-plugins.bat enable rabbitmq_federation_management

Команды активации плагинов необходимо выполнить для всех экземпляров RabbitMQ, участвующих в федеративном обмене.

Получение списка активных плагинов выполняется командой:

ü  Linux

rabbitmq-plugins list

ü  Windows

rabbitmq-plugins.bat list

4 Настройка политик (policies)

4.1 Исходные данные

Отправка сообщений в восходящий поток осуществляется на основе политик (Policies). Управление политиками осуществляется с помощью пункта меню «Admin → Policies» веб-консоли управления RabbitMQ.

Штатно комплекс РСДУ использует в своей работе точки обмена и очереди, название которых начинается на «rsdu.». Поэтому, в рамках настоящего документа, представлены инструкции по настройке политик для всех объектов RabbitMQ, имя которых начинается с «rsdu.».

Возможно использование точек обмена и очередей с иным правилом наименований. В этом случае, необходимо настроить политики, работающие с этими объектами RabbitMQ. Процедура настройки таких политик аналогична представленной в настоящем руководстве.

Для работы технологии Federation необходимо создать одинаковые политики на всех экземплярах RabbitMQ, участвующих в федеративном обмене.

4.4 Изменение параметров политики

Изменение параметров политик осуществляется с помощью пункта меню «Admin → Policies → Add/update a policy» веб-консоли управления RabbitMQ, по аналогии с созданием новой политики (см. «Создание политики»).

Если в поле Name указано имя созданной ранее политики, то новые параметры будут применены к этой политике. Если имя будет отличным от ранее созданных политик, то выполнится создание новой политики с указанными параметрами.

 

5 Контроль работы Federation

5.1 Контроль состояния восходящих потоков

Текущее состояние восходящих потоков отображено в таблице пункта меню «Admin → Federation Status» веб-консоли управления RabbitMQ (Рисунок 7).

Поле Statе таблицы «Running Links» содержит информацию о текущем состоянии восходящих потоков. Работающие потоки отмечены состоянием «running».

Кнопка «Restart» выполняет перезапуск восходящего потока.

5.2 Контроль соединений

Для передачи сообщений Federation создает два типа соединений:

ü  внешнее соединение с другим узлом по протоколу AMQP;

ü  внутреннее соединение (Direct).

Контроль текущих активных соединений выполняется с помощью пункта меню «Connection» (Рисунок 8).

Контролируется состав текущих соединений и состояние соединений по протоколу AMQP с другими экземплярами RabbitMQ.

Для каждого соединения RabbitMQ создает канал (Channel). Контроль каналов выполняется с помощью пункта меню «Channels».

Необходимо проконтролировать состав и состояние соединений и каналов, проверить адреса и наименования виртуальных хостов, используемых при подключении.

5.3 Контроль точек обмена и очередей

У всех точек обмена (Exchanges) RabbitMQ, для которых настроена политика Federation, в поле Features содержится название используемой политики (Рисунок 9).

Необходимо проверить у точек обмена наименование политики, используемой в федеративном обмене.

Дополнительно, для работы Federation, создаются точки обмена (Exchanges) и очереди (Queues) для внутренних нужд плагина Federation. Такие объекты в имени содержат «federation:» (Рисунок 10).

Необходимо проверить состав и параметры дополнительных точек обмена Federation. Состав дополнительных точек обмена и очередей должен соответствовать составу объектов, участвующих в федеративном обмене.

Приложение А. Параметры восходящих потоков

ü  Name – наименование канала.

Для удобства рекомендуется в имени восходящего потока указывать наименование виртуального хоста.

ü  URI – строка подключения к другому экземпляру RabbitMQ.

Формат строки подключения:

amqp://user:password@server-addr/vhost

где:

o   user и password – имя пользователя и пароль для подключения к экземпляру RabbitMQ;

o   server-addr – сетевое имя или IP-адрес сервера экземпляра RabbitMQ;

o   vhost – наименование виртуального хоста.

ü  Prefetch count – максимальное количество не подтверждённых сообщений, которые могут быть помещены в канал связи за раз. По умолчанию - 1000.

ü  Reconnect delay – время ожидания в секундах перед попыткой восстановления канала связи после разрыва. По умолчанию - 5 секунд.

ü  Acknowledgement Mode – режим подтверждения сообщений при передаче по каналу связи. Возможные значения:

o   on-confirm (по умолчанию) – подтверждение выполняется по факту доставки сообщения в другой экземпляр RabbitMQ;

o   on-publish – подтверждение выполняется по факту отправки сообщения в канал связи (при условии активности канала связи);

o   no-ack – подтверждение не используется.

ü  Trust User-ID – использовать доверительного пользователя. Заполнение поле user-id сообщения при ретрансляции. По умолчанию No.

ü  Exchange – название точки обмена, используемой на втором экземпляре для работы Federation. Если не задана, то используются имена точек обмена текущего экземпляра, для которых настроена политика (policy).

ü  Max hops – максимальное количество узлов федерации в цепочке передачи сообщений. По умолчанию - 1.

ü  Expires – время в миллисекундах жизни потока передачи сообщений. Если поле пустое, то время не ограничено.

ü  Message TTL – время в миллисекундах жизни сообщения. Если поле пустое, то время жизни не ограничено.

ü  HA Policy – определяет параметр «x-ha-policy» для очереди потока для федеративного обмена. Значение по умолчанию – «none», что означает, что очередь не является HA.

ü  Queue – имя очереди, используемое на втором экземпляре для федеративного обмена. Если поле не задано, то используются имена очередей, для которых настроена политика (policy).
