1  Термины, определения, обозначения и сокращения

В настоящем документе приняты следующие термины, определения, обозначения и сокращения:

API (Application Programming Interface) – интерфейс программирования приложений, интерфейс прикладного программирования. Набор готовых классов, процедур, функций, структур и констант, предоставляемых приложением (библиотекой, сервисом) для использования во внешних программных продуктах;

MRID – уникальный идентификатор информационных объектов справочников и журналов, используемый в СК-2007;

ODBC (Open Database Connectivity) – программный интерфейс (API) доступа к базам данных, разработанный фирмой Microsoft;

SOAP (Simple Object Access Protocol) – протокол обмена структурированными сообщениями в распределённой вычислительной среде. Первоначально SOAP предназначался в основном для реализации удалённого вызова процедур (RPC);

SQLite – компактная встраиваемая реляционная база данных, используемая для хранения локальных данных;

URL (Uniform Resource Locator) – единообразный определитель местонахождения (адрес) ресурса в вычислительной сети;

БДТИ – База Данных Технологической Информации РСДУ;

ГОУ (Групповой объект управления) – совокупность одной или нескольких групп точек поставки (ГТП), в отношении которой Системным оператором отдаются и фиксируются команды на изменение режима работы;

ГТП (Группа точек поставки) – одна или несколько точек поставки, относящихся к одному узлу расчетной модели и/или к единому технологически не делимому энергетическому объекту и ограничивающих территорию, в отношении которой купля-продажа электрической энергии (мощности) на оптовом рынке осуществляется только данным участником, и используемых для определения и исполнения участником оптового рынка обязательств, связанных с поставкой и оплатой электрической энергии (мощности). Группа точек поставки определяется Системным и Коммерческим операторами в соответствии с Положением о порядке получения статуса субъекта оптового рынка и ведения реестра субъектов оптового рынка электрической энергии и мощности;

КДУ – Команда диспетчерского управления;

ОИК – оперативно-информационный комплекс;

Сервис EMSDataService – веб-сервис на базе SOAP, предоставляющий функции управления журналом КДУ в СК-2007;

СК-2007 – универсальная многофункциональная система для построения автоматизированных систем коммерческой и технологической диспетчеризации в сетевых и генерирующих компаниях;

СО (СО ЕЭС) – Системный Оператор Единой Энергосистемы Российской Федерации.

 

2 Общие сведения

2.1 Назначение

Сервис предназначен для ретрансляции КДУ в темпе их регистрации из системы РСДУ в СК-2007.

Сервис использует интерфейс (API) EMSDataService СК-2007, реализованный в виде веб-сервиса SOAP.

2.2 Необходимые условия

Перед переходом к действиям, описанным ниже в руководстве, должны быть выполнены следующие необходимые условия:

ü  установлена ОС Microsoft Windows 2008 или выше;

ü  сервер должен быть включен в домен, для пользователей которого настроена система санкционирования СК-2007;

ü  установлен пакет .NET Framework 4.5;

ü  установлен пакет Visual C++ 2010 Redistributable Package;

ü  установлен пакет клиента Oracle 11.2;

ü  настроен источник ODBC для БДТИ РСДУ;

ü  выполнены обновления в БДТИ РСДУ:

o   2013_12_02_DG_NEW;

o   2014_03_04_J_DGSTATE;

o   2015_10_08_ORG_STRUCTURE_AST;

o   2015_09_15_DG_KDU_EXTENTION.

Необходимо наличие следующих исходных данных:

ü  справочник ГОУ СК-2007 с MRID;

ü  справочник центров диспетчерского управления СК-2007 с MRID.

2.3 Необходимые знания

Для выполнения действий, описанных ниже в руководстве, требуются следующие знания:

ü  имя пользователя ОС, имеющего права администратора и его пароль;

ü  имя пользователя и пароль в домене Windows с правами на доступ к сервису EMSDataService СК-2007;

ü  URL основного и резервного серверов СК-2007, на которых настроен сервис EMSDataService (как правило, в СК-2007 сервис EMSDataService работает на серверах ОИК №1 и ОИК №2).

2.4 Ссылки

В руководстве используются ссылки на следующую документацию к комплексу РСДУ:

ü  «Руководство администратора. Описание серверов и точек доступа в комплексе РСДУ»;

ü  «Руководство администратора. Описание структуры БД РСДУ».

4 Конфигурирование сервиса

Для конфигурирования сервиса используется приложение TransferKDUtoSK2007.Config.exe, расположенное в каталоге установки сервиса (по умолчанию – каталог C:\RsduServices\RSDU.INP\TransferKDUtoSK2007).

Рисунок 2 - Интерфейс приложения настройки сервиса вкладка «Настройки»

Окно конфигуратора содержит следующие вкладки:

ü  Настройки – общие настройки сервиса, такие как параметры подключения к системам, периоды опроса, часовые пояса и т.д.

ü  Связь справочников – содержит описание элементов справочников интегрируемых систем и связей между ними.

ü  Рабочие переменные – это переменные, которые сохраняются при остановке сервиса и используются при его старте; рабочие переменные могут быть отредактированы.

ü  Администрирование очереди передачи КДУ – содержит инструменты для контроля разбора очереди КДУ на передачу в СК-2007.

Конфигурация сервиса хранится в файлах БД SQLite в каталоге db – «config.db» и «work.db».

Файл «config.db» – файл конфигурации сервиса; содержит общие настройки сервиса (вкладка Настройка) и описание справочников интегрируемых систем (вкладка Связь справочников).

Файл «work.db» – файл для хранения рабочих переменных и очереди передачи КДУ.

Для удобства выполнения настройки необходимо дополнительно запустить приложение «Навигатор БД» комплекса РСДУ (запустить «Панель оператора» и вызвать приложение «Навигатор БД»). С помощью навигатора БД, например, можно будет получить информацию об идентификаторах объектов, используемых при настройке связей между системами.

4.2 Настройка связей справочников

Вкладка «Связь справочников» содержит описание справочников СК-2007 и РСДУ, а также связей между их элементами.

Для работы сервиса необходимо заполнить следующие справочники:

ü  справочник КДУ – содержит описание команд диспетчерского управления, инициатив, причин отклонений и комментариев к причинам;

ü  справочник ГОУ и персонала – содержит описание ГОУ и персонала, выполняющего получение и регистрацию КДУ на станции; как правило, КДУ получает начальник смены станции;

ü  справочник диспетчерских центров – содержит идентификаторы диспетчерских центров СО, выдающих КДУ.

4.2.2 Справочник КДУ

Справочник КДУ содержит описание команд, инициатив, причин отклонений и шаблонов комментариев к причинам отклонений. Справочник строится на базе справочника СК-2007.

Соответствующие элементы справочника должны содержать идентификаторы аналогичных элементов в системах СК-2007 и РСДУ.

Справочник КДУ является условно статичным и может меняться в следующих случаях:

ü  изменение регламента СО, связанного с регистрацией команд диспетчерского управления, сообщений и т.д.;

ü  изменение уникальных идентификаторов (MRID) элементов справочника на стороне СК-2007.

внимание!

Дистрибутив сервиса содержит преднастроенный справочник КДУ. Редактирование справочника необходимо выполнять только в случае возникновения ошибок передачи КДУ, связанных с сопоставлением элементов справочника.

 

4.2.2.1 Описание структуры справочника

Справочник оперирует следующими типами элементов:

ü  Группа команд (TYPE_KDU_GROUP) – определяет группу команд, разрешений и сообщений.

ü  Команда (TYPE_KDU_CMD) – определяет команду.

ü  Инициатива (TYPE_KDU_INITIATIVE) – определяет инициативу.

ü  Причина отклонения (TYPE_KDU_CAUSE) – определяет причину отклонения.

ü  Комментарий к причине (TYPE_KDU_CAUSE_COMMENT) – определяет шаблон комментария к причине отклонения.

Каждый элемент справочника содержит идентификатор соответствующего элемента в СК-2007 и РСДУ.

В СК-2007 элементы справочника определяются уникальным идентификатором MRID.

В РСДУ элементы справочника КДУ содержатся в соответствующих таблицах в БДТИ раздела DG (см. описание раздела DG в документации БДТИ РСДУ) и определяются полем ID таблиц.

Кроме идентификаторов записей таблиц БД РСДУ (поле ID) справочник поддерживает специализированное символьное обозначение – «DEFAULT».

Обозначение «DEFAULT» определяет элемент справочника как элемент по умолчанию при поиске соответствия. Таким образом, если по идентификатору РСДУ не найден элемент в «Справочнике КДУ» заданного типа, то будет использоваться тот элемент, у которого в поле «Идентификатор РСДУ» задано обозначение «DEFAULT». Это правило распространяется на элементы определенного типа и на определенном уровне вложенности. Например, первый уровень иерархии справочника содержит «Группы команд», – на этом уровне может быть задан один элемент как «DEFAULT». 4.2.2.2 Заполнение справочника

Приложение настройки сервиса поддерживает автоматизированное заполнение справочника КДУ на базе справочников СК-2007.

При этом приложение выполняет подключение к серверу СК-2007 сервису EMSDataService (используя параметры подключения, заданные во вкладке «Настройки»), производит запрос информационной модели и на базе полученных данных заполняет «Справочник КДУ». В справочнике будет выполнено формирование полного дерева поддерживаемых СК-2007 команд, разрешений и сообщений.

Дальнейшая привязка элементов сформированного справочника производится вручную и заключается в заполнении поля «Идентификатор РСДУ».

4.2.3 Справочник ГОУ и персонала

Справочник ГОУ и персонала содержит описание групповых объектов управления и персонала, имеющего право на регистрацию команд диспетчерского управления. Справочник строится на базе справочника СК-2007.

4.2.3.1 Описание структуры справочника

Справочник оперирует следующими типами элементов:

ü  группа или каталог (TYPE_GOU_GROUP) – используется для каталогизации справочника.

ü  групповой объект управления (ГОУ) (TYPE_GOU) – определяет Групповой объект управления.

ü  персонал станции (TYPE_GOU_USER) – определяет персонал станции, имеющий право получения и регистрации КДУ для данного ГОУ.

В РСДУ элементы справочника содержатся в соответствующих таблицах в БДТИ общего системного раздела (таблица S_USERS) и раздела DG (см. описание разделов в документации «Описание структуры БД РСДУ. Руководство администратора»). Элементы определяются полем ID таблиц.

Таким образом, если по идентификатору РСДУ не найден элемент заданного типа, то будет использоваться тот элемент, у которого в поле «Идентификатор РСДУ» задано обозначение «DEFAULT». Это правило распространяется на элементы типа «Персонал станции».

5.    По завершению ввода нажать кнопку «Сохранить» ().

 

5 Администрирование сервиса

5.1 Регистрация и удаление сервиса в ОС Windows

5.1.1 Описание сервиса  (файл ServiceDefinition.xml)

Описание сервиса, необходимое для регистрации сервиса в консоли управление Службы Windows находится в файле «ServiceDefinition.xml».

Файл «ServiceDefinition.xml» располагается в каталоге установки сервиса, по умолчанию C:\RsduServices\RSDU.INP\TransferKDUtoRSDU.

<?xml version="1.0" encoding="utf-16" ?>

<ServiceDefinition

xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"

xmlns:xsd="http://www.w3.org/2001/XMLSchema"

xmlns="rsdu.ema">

  <Name> RSDUINPTransferKDUtoSK2007</Name>

  <Display> RSDU.INP> Сервис передачи КДУ в СК-2007</Display>

  <Description> Сервис передачи КДУ в СК-2007. Входит в состав подсистемы RSDU.INP комплекса РСДУ.</Description>

</ServiceDefinition>

 

Описание тэгов файла «ServiceDefinition.xml»:

ü  Name – уникальное в рамках ОС название сервиса; как правило, в названии сервиса используются латинские буквы и цифры;

ü  Display – название сервиса, которое будет отображаться в консоли управления Службы Windows; допускается использование русских букв;

ü  Description – текстовое описание сервиса.

5.1.2 Установка сервиса

Необходимо войти в операционную систему Windows пользователем с правами администратора и далее:

1.    Скопировать дистрибутив сервиса (каталог RSDU.INP\TransferKDUtoSK2007) на целевой хост в каталог C:\RsduServices\RSDU.INP.

2.    Выполнить установку сервиса с помощью скрипта «01. Install.cmd»: открыть командную строку Windows от имени Администратора, перейти в каталог размещения сервиса и выполнить скрипт «01. Install.cmd».

После выполнения скрипта «01. Install.cmd» сервис должен отображаться в консоли управления Службами Windows (Services).

Название сервиса Windows: «RSDU.INP> Сервис передачи КДУ в СК-2007».

3.    Выполнить конфигурирование сервиса с помощью приложения TransferKDUtoSK2007.Config.exe (см. раздел настоящей инструкции «Конфигурирование сервиса»);

4.    Выполнить настройку пользователя, от имени которого сервис будет запускаться в ОС Windows.

Для обеспечения доступа к сервису EMSDataService СК-2007 сервис передачи КДУ должен запускаться от имени доменного пользователя, которому в системе санкционирования СК-2007 назначены соответствующие права.

внимание!

Сервер, на котором выполняется настройка сервиса передачи КДУ, должен быть включен в домен, пользователи которого настроены в системе санкционирования СК-2007.

 

Для настройки доменного пользователя и системы санкционирования СК-2007 необходимо обратиться к администратору системы СК-2007 и/или документации производителя на систему СК-2007.

Для настройки пользователя необходимо открыть свойства сервиса «RSDU.INP> Сервис передачи КДУ в СК-2007» в консоли управления Службами Windows (Services), перейти во вкладку «Вход в систему» и заполнить следующие поля:

ü  установить галочку «С учетной записью»;

ü  указать домен и имя пользователя;

ü  указать пароль с подтверждением;

ü  по завершению ввода нажать «ОК».

5.    Выполнить настройку автоматического запуска сервиса: открыть свойства сервиса «RSDU.INP> Сервис передачи КДУ в СК-2007» в консоли управления Службами Windows (Services), перейти во вкладку «Общие» и указать тип запуска «Автоматически».

6.    Выполнить настройку автоматического восстановления сервиса: настройка параметров восстановления при сбоях выполняется во вкладке «Восстановление». При этом необходимо указать для Первого, Второго и Последующих сбоев перезапуск через 1 минуту.

5.1.3 Удаление сервиса

Необходимо войти в операционную систему Windows пользователем с правами администратора и далее:

1.    Выполнить скрипт удаления сервиса «04. Uninstall.cmd»: открыть командную строку Windows от имени Администратора, перейти в каталог размещения сервиса (по умолчанию C:\RsduServices\RSDU.INP\TransferKDUtoSK2007) и выполнить скрипт «04. Uninstall.cmd».

2.    В консоли управления Службами Windows (Services) проверить отсутствие в списке сервиса «RSDU.INP> Сервис передачи КДУ в СК-2007».

3.    При необходимости удалить каталог размещения сервиса.

5.2 Запуск и останов сервиса

Запуск и останов сервиса можно выполнить следующими способами:

ü  с помощью консоли управления службами Windows (Services → Службы);

ü  из командной строки Windows.

5.2.1 Запуск/останов сервиса из командной строки

Запуск и останов сервиса из командной строки выполняется с помощью скриптов «02. Start.cmd» и «03. Stop.cmd» соответственно.

Для работы скриптов необходимы права администратора.

Для запуска сервиса необходимо выполнить следующие шаги:

1.    Открыть командную строку Windows с правами администратора: вызвать контекстное меню для приложения «Командная строка» (Пуск → Все программы → Стандартные → Командная строка) и выбрать пункт «Запуск от имени администратора».

2.    Перейти в каталог установки сервиса.

По умолчанию каталог установки: C:\RsduServices\RSDU.INP\TransferKDUtoSK2007.

3.    Запустить скрипт «02. Start.cmd».

Работу сервиса можно проконтролировать по сообщениям в лог-файле.

Для остановки сервиса необходимо выполнить следующие шаги:

1.    Открыть командную строку Windows с правами администратора и перейти в каталог установки сервиса (см. выше).

2.    Запустить скрипт «03. Stop.cmd».

5.2.2 Запуск/останов из консоли управления службами Windows

Для запуска или остановки сервиса с использованием консоли управления службами Windows необходимо выполнить следующие операции:

1.    Открыть консоль управления службами Windows («Пуск → Панель  управления → Администрирование → Службы»).

2.    В списке служб найти и выделить службу «RSDU.INP> Сервис передачи КДУ в СК-2007».

3.    Для запуска службы нажать кнопку «Запуск службы».

4.    Для остановки  службы нажать кнопку «Остановка службы».

Текущее состояние службы отображается в колонке «Состояние» в списке служб.

Работу сервиса можно проконтролировать по сообщениям в лог-файле (см. раздел Контроль лог-файлов).

5.4 Администрирование очереди передачи КДУ

Вкладка «Администрирование очереди передачи КДУ» отображает элементы очереди на передачу, состояние ее элементов и сообщения об ошибках.

Рисунок 15 - Интерфейс консоли администрирования очереди на передачу КДУ

5.4.1 Описание элементов управления

Поле «Статус» определяет фильтр элементов очереди по текущему состоянию.

Возможные варианты поля «Статус»:

ü  ALL – все записи очереди.

ü  DONE – успешно обработанные (переданные) КДУ.

ü  ERROR – КДУ, при передаче которых возникла ошибка.

ü  NEW – новые КДУ в очереди, ожидающие передачу.

Поле «Дата начала» и «Дата окончания» задают фильтр выбора элементов очереди по времени поступления в очередь – фильтр по полю «Дата» таблицы.

Кнопка «Выбрать» производит выбор элементов очереди по заданному полями «Статус», «Дата начала» и «Дата окончания» фильтру.

Кнопка «Удалить» производит удаление выбранного элемента очереди в таблице.

Кнопка «Изменить» открывает окно изменения статуса и сообщения об ошибке элемента очереди.

Кнопка «Очистить» позволяет выполнять очистку старых обработанных записей очереди. Используется для уменьшения размера файла «Work.db».

5.4.2 Описание полей таблицы отображения элементов очереди

Поля таблицы отображения элементов очереди:

ü  Id – идентификатор элемента очереди;

ü  IdParent – ссылка на другой элемент очереди; используется при передаче операций над зарегистрированными КДУ – операция КДУ;

ü  Дата – дата поступления КДУ в очередь на передачу;

ü  Состояние – текущее состояние элемента в очереди; возможны состояния: NEW – новый элемент, DONE – успешно обработанный элемент, ERROR – элемент, при обработке которого возникла ошибка;

ü  Id объекта (локальный) – идентификатор КДУ в РСДУ;

ü  Id объекта (внешний) – идентификатор (MRID) КДУ в СК-2007;

ü  Текст – текстовое представление КДУ.

Для удобства контроля процесса передачи КДУ под таблицей расположены два окна отображающих текстовое представление КДУ и в виде XML.

XML представление КДУ позволяет получить расширенную информации об идентификаторах элементов справочников, используемых при регистрации КДУ.

Пример XML представления КДУ:

<?xml version="1.0" encoding="utf-16"?>

<RsduKDU xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xmlns:xsd="http://www.w3.org/2001/XMLSchema">

  <Id>2856</Id>

  <KeyUnique>2739c626-374e-8b4c-9781-a9472e96252e</KeyUnique>

  <IdCommand>1</IdCommand>

  <IdObject>3</IdObject>

  <dt1970_start>1459176540</dt1970_start>

  <dt1970_end>1459176600</dt1970_end>

  <Value>300</Value>

  <IdInitiative>1</IdInitiative>

  <IsDeleted>false</IsDeleted>

  <dt1970_create>1459176606</dt1970_create>

  <IdUserCreate>1</IdUserCreate>

  <dt1970_modify xsi:nil="true" />

  <IdUserModify xsi:nil="true" />

  <dt1970_receive>1459176540</dt1970_receive>

  <sComment>Нагрузка станции (наименование) ниже планового ДГ на «N» МВт</sComment>

  <IdCause>3</IdCause>

  <IdCauseCommentTmpl>3</IdCauseCommentTmpl>

  <IdSourceCnt>3000022</IdSourceCnt>

  <IdSourceDisp>24</IdSourceDisp>

  <TextCommand>Дата: 28.03.2016 19:49

ГОУ: ОЛИМПГРЭС ГТП 1

Команда: ИВ, Генерация 300МВт

Период: 28.03.2016 с 19:49 к 19:50

Причина отклонения: Неплановое изменение генерации

Нагрузка станции (наименование) ниже планового ДГ на «N» МВт

Отдано: Красноярское РДУ

Отдал: Диспетчер (Красноярское РДУ)

Принял: Администратор комплекса РСДУ5

</TextCommand>

</RsduKDU>

 

Описание тэгов и атрибутов XML представления:

ü  Id – идентификатор КДУ в журнале в таблице DG_KDU_JOURNAL;

ü  IdCommand – идентификатор команды из таблицы DG_KDU_COMMAND;

ü  IdInitiative – идентификатор инициативы из таблицы DG_KDU_INITIATIVE;

ü  IdCause – идентификатор причины отклонения из таблицы DG_KDU_CAUSE;

ü  IdCauseCommentTmpl – идентификатор шаблона комментария к причине отклонения из таблицы DG_KDU_COMMENT_TMPL;

ü  KeyUnique – глобальный идентификатор КДУ в журнале;

ü  IdObject – идентификатор ГОУ из таблицы DG_GROUPS;

ü  dt1970_start и dt1970_end – даты начала и окончания КДУ в формате DT1970;

ü  Value – значение (определяет значение МВт, указанное при регистрации КДУ);

ü  IsDeleted – флаг, определяющий удаление КДУ;

ü  dt1970_create – дата регистрации КДУ в РСДУ в формате DT1970;

ü  IdUserCreate – идентификатор пользователя РСДУ, зарегистрировавшего КДУ;

ü  dt1970_modify – дата изменения КДУ в формате DT1970;

ü  IdUserModify – идентификатор пользователя РСДУ, выполнившего изменение КДУ;

ü  dt1970_receive – дата получения КДУ в формате DT1970;

ü  sComment – комментарий;

ü  IdSourceCnt – идентификатор диспетчерского центра из таблицы AST_CNT, отдавшего КДУ;

ü  IdSourceDisp – идентификатор персонала из таблицы AST_PERSONNEL, отдавшего КДУ;

ü  TextCommand – текстовое представление КДУ.

5.5 Контроль лог-файлов

Лог-файл сервиса располагается в каталоге установки сервиса и называется «TransferKDUtoSK2007.Service.log».

Например,

C:\RsduServices\RSDU.INP\TransferKDUtoSK2007\ TransferKDUtoSK2007.Service.log

Раз в неделю сервис выполняет архивирование и очистку лог-фала. Архивный лог-файл сжимается и копируется в каталог archivelogs. В имени архивированного лог-файла содержится дата архивации.

Например,

C:\RsduServices\RSDU.INP\TransferKDUtoSK2007\archivelogs\ TransferKDUtoSK2007.Service_2016-01-07.log.gz
